Overview Page Restructure Plan

Commerce Best Sellers Plugin · March 2026 · Based on user feedback

1. Problem Statement

The current overview page is organized by data type (Orders, Customers, Products) rather than by the questions a store owner is actually asking. The top row buries the most important business health signals among secondary metrics like Total Discount and Items Sold.

2. Current vs. Proposed Layout

Current Layout

Orders (6 KPI cards)
Revenue, Orders, AOV, Total Discount, Items Sold, Avg Items/Order + Revenue/Orders/AOV toggle chart
↓
Customers (3 KPI cards)
Customers, New Customers, Repeat Rate + New vs Returning chart + Credentialed vs Guest table (side-by-side) + Cart Abandonment widget
↓
Products (2 KPI cards)
Unique Products Sold, Product Revenue + Best Sellers table

Proposed Layout

Health Check (hero row)
Revenue, Orders, AOV, Repeat Rate — the four metrics that answer "how's the business doing right now?"
↓
Revenue & Orders
Total Discount, Items Sold, Avg Items/Order (secondary cards) + Revenue/Orders/AOV toggle chart with period comparison
↓
Customers & Retention
Customers, New Customers, LTV (new top-level stat) + Credentialed vs Guest table (promoted to full-width) + New vs Returning chart
↓
Cart Abandonment
Own section (separated from Customers). Keep existing widget as-is.
↓
Product Performance
Unique Products Sold, Product Revenue + Best Sellers table

4. Implementation Steps

Step 1: Restructure cardGroups() in OverviewController

File: src/controllers/OverviewController.php (lines 154-168)

Change the cardGroups() method from the current Orders/Customers grouping to:

'Health Check' => [
    'keys' => ['revenue', 'orders', 'aov', 'repeatRate'],
    // No link -- this is the summary row, not a drill-down
],
'Revenue & Orders' => [
    'keys' => ['totalDiscount', 'itemsSold', 'avgItemsPerOrder'],
    'link' => 'best-sellers/orders',
    'linkLabel' => 'See all orders',
],
'Customers & Retention' => [
    'keys' => ['customers', 'newCustomers'],
    'link' => 'best-sellers/customers',
    'linkLabel' => 'See all customers',
],

repeatRate moves up to Health Check. The LTV card is built manually in the controller (same pattern as the Products group) since it comes from ltvComparison, not PeriodStats.

Step 2: Fetch previous-period LTV and build the LTV card

File: src/controllers/OverviewController.php

Add the previous-period LTV fetch, then manually build the LTV KPI card and append it to the Customers & Retention group. Follow the same pattern used for Products cards (lines 66-84).

// Fetch previous LTV
$prevLtvComparison = $customerStats->getLtvComparison(
    $dateRange->getPrev()->fromDT,
    $dateRange->getPrev()->toDT
);

// Compute blended average LTV
$totalCustomers = $ltvComparison->credentialed->count + $ltvComparison->guest->count;
$avgLtv = $totalCustomers > 0
    ? ($ltvComparison->credentialed->totalRevenue + $ltvComparison->guest->totalRevenue) / $totalCustomers
    : 0;

$prevTotalCustomers = $prevLtvComparison->credentialed->count + $prevLtvComparison->guest->count;
$prevAvgLtv = $prevTotalCustomers > 0
    ? ($prevLtvComparison->credentialed->totalRevenue + $prevLtvComparison->guest->totalRevenue) / $prevTotalCustomers
    : 0;

Then add the LTV card to the Customers & Retention cards array.

Step 3: Update controller to pass named section data

File: src/controllers/OverviewController.php

Instead of relying on label-string matching in the template, pass named variables for each section. The current template checks group.label == 'Orders', group.label == 'Customers', etc. which is fragile. Switch to explicit template variables: 

return $this->renderTemplate('best-sellers/_overview', [
    // Health Check
    'healthCheckCards' => [...],

    // Revenue & Orders
    'revenueOrdersCards' => [...],
    // (chart data stays the same: dailyLabels, dailyOrders, etc.)

    // Customers & Retention
    'customerCards' => [...],
    'newVsReturning' => $newVsReturning,
    'ltvComparison' => $ltvComparison,

    // Cart Abandonment (unchanged)
    'cartAbandonment' => $cartAbandonment,

    // Products (unchanged)
    'productCards' => [...],
    'bestSellers' => $bestSellers,
    'bestSellerElements' => $bestSellerElements,

    // Sparklines, chart data, etc.
    ...
]);

Step 5: Split customer-charts.twig into two widget templates

File: src/templates/_includes/widgets/customer-charts.twig (split into two)

Currently this template renders the New vs Returning chart and the Credentialed vs Guest table side-by-side in a .bs-columns layout. Split into:

  • _includes/widgets/new-vs-returning-chart.twig — just the chart canvas and container
  • _includes/widgets/ltv-table.twig — the Credentialed vs Guest table, now full-width

The Credentialed vs Guest table is promoted above the chart. It renders at full width instead of half width, giving the LTV and ordering behavior data more visual prominence. This is especially valuable for B2B stores where the credentialed vs guest gap is a key insight.

6. Notes from User Feedback

  • Avg Items/Order (35.60) — interesting for B2B (Trufast, knife bolts, mower blades) but not a top-level metric. Moved to secondary stat inside Revenue & Orders section.
  • Credentialed vs Guest table — one of the most valuable widgets for B2B. Shows the LTV and ordering behavior gap. Promoted to full-width within Customers & Retention, placed above the New vs Returning chart.
  • Revenue chart spike annotations — noted as a future feature consideration, not part of this restructure. Would benefit from a "notable events" callout for spikes like the Mar 13-14 jump to ~$70k.
